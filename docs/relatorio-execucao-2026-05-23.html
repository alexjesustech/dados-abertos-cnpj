Decisões registradas

DecisãoEscolhaMotivo
Framework HTTPFastAPI 0.115+Pydantic 2 nativo — mesma definição alimenta MCP via FastMCP; Swagger em pt-BR grátis; TestClient zero-fricção.
Acesso ao bancosqlite3 puro + repositórios finosSchema é fixo (vem da RFB), queries são poucas. ORM (SQLAlchemy) somaria 200 ms de import e zero benefício.
Concorrência leitura/escritaURI ?mode=roDefesa real contra INSERT acidental; WAL (configurado pelo pipeline) permite N readers + 1 writer.
Cache de lookupsCarregado no lifespan startup~30 MB RAM, ~200 k entradas, traduções O(1). Reduz hit no SQLite em quase todas as rotas.
Cache de /statsLazy em app.stateCOUNT(*) em 68M+ linhas sem ANALYZE leva ~60 s. Primeiro hit popula; restantes são instantâneos.
Período da safraSUBSTR(MAX(data_importacao), 1, 7)controle_importacao não armazena o período da RFB; deriva da data de import. Heurística boa o bastante para uso local.
Gestão de depsuv + pyproject.tomlResolução em ms, lockfile reproduzível, entrypoints sem ativar venv. requirements.txt mantido auto-gerado.
MCP integra com a APIImporta app.servicos diretoSem rede, sem subprocess, mesma transação sqlite, mesmo Pydantic. Single source-of-truth para "CNPJ completo".
Paginação no MCPManual limit/offset + tem_maisMCP não tem paginação nativa para tools/call — tools sem disciplina estouram o contexto do LLM.
CPF de sócio PFExpor como vem (***NNNNNN**) + flag mascarado: trueRFB sempre publica PF mascarado em dados abertos. Não tentar desmascarar.
Validador alfanuméricoMesmo algoritmo para os dois formatosO CNPJ legado é caso particular do novo: ord('0') − 48 = 0. Um validador único bastou.
06

Pontos de atrito resolvidos

i
/periodo-atual devolvia "Socios9"
onde: app/rotas/meta.py causa: SUBSTR(arquivo, 1, 7) impacto: baixo
A query lia o nome do arquivo (Socios9.zip), não o período. Correção: deriva de SUBSTR(MAX(data_importacao), 1, 7) → "2026-05".
ii
/stats levava 62 segundos no primeiro hit
onde: app/rotas/meta.py causa: COUNT(*) sem ANALYZE impacto: médio
COUNT em tabelas de 68M+ linhas sem sqlite_stat1 populado. Correção dupla: cache lazy em app.state.stats_cache + ANALYZE em todas as tabelas-fato na migration. Hoje primeiro hit roda em ~1 s; cache hit é instantâneo.
iii
cnaes_por_municipio em Brasília levava 70 segundos
onde: mcp_server/tools/cnaes_por_municipio causa: GROUP BY sem índice composto impacto: alto
GROUP BY em todos os estabelecimentos de Brasília (mais de 1.000 CNAEs distintos) sem índice (municipio, cnae_fiscal_principal) fazia table-scan. Correção: CREATE INDEX idx_estab_municipio_cnae na migration. Speedup mensurado: 1.000×.
iv
vinculos_pj sem índice em cnpj_cpf_socio
onde: app/repositorios/socio_repo.py causa: tabela socios sem PK declarada impacto: alto
Sem índice, busca por CPF mascarado escaneia 27,6M linhas. Correção: CREATE INDEX idx_socios_documento + idx_socios_basico_id. Tool agora responde em < 1 ms.

O que não fazer agora

  • Não expor a API publicamente sem auth. Default é 127.0.0.1:8000 por uma razão — auth + rate-limit ficam no Caminho 04.
  • Não tentar diff de safras com snapshot atual. delta_mensal está em MVP (só metadados); diff real exige preservar tabelas históricas — Caminho 02.
  • Não migrar database.py para SQLAlchemy. Schema é fixo, queries são poucas. Repositórios em sqlite3 puro bastam.
  • Não fazer busca reversa por nome de sócio PF indexável. O briefing Nº 001 marcou como zona LGPD vermelha — Casa dos Dados sofre pedidos recorrentes.
  • Não tentar tools que retornem mais de 500 linhas em uma única chamada. Disciplina de paginação é parte do contrato — tools que não caibam viram resources, não tools.
