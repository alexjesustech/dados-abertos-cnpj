CAMINHO 01 · A + I + J

"Caixa-preta de CNPJ pra mim" — plano executável

Três artefatos — validador, API local e MCP server — em três sprints úteis. Stack: FastAPI sobre sqlite3 puro, uv gerindo deps, FastMCP expondo nove ferramentas tipadas. Começa por J (validador alfanumérico) porque a vigência do CNPJ alfa é 06/07/2026 e o algoritmo é dependência limpa de A e I. Total: três semanas e meia úteis, com margem de buffer dentro do orçamento de quatro do briefing Nº 001.

04

Artefato J · Validador CNPJ

J
cnpj_lib/ · biblioteca standalone
deps: stdlib esforço: P publishable: sim vigência: 06/07/2026

Cobre validação e geração de DV para os dois formatos. Algoritmo é único — o CNPJ antigo (14 dígitos) é caso particular do novo, pois ord('0') − 48 = 0. Não há conversão entre os universos; um CNPJ alfanumérico é necessariamente novo, emitido após a vigência.

Arquivos

  • validador.py — validar(cnpj: str) → bool, calcular_dv(base12: str) → str, eh_alfanumerico(s) → bool
  • formatador.py — normalizar, formatar (XX.XXX.XXX/XXXX-DD), fragmentar(cnpj14) → (basico, ordem, dv), mascarar_cpf
  • dominio.py — dicionários hardcoded para enums ausentes no BD

Critérios de aceite

  • Casos da NT 2025.001 passam: 12.ABC.345/01DE-35, AB12CD34/EFGH-83.
  • 10 CNPJs antigos famosos (Itaú, BB, Petrobras...) — todos válidos.
  • Hypothesis: validar(base + calcular_dv(base)) retorna True para 1.000 bases aleatórias.
  • Coverage ≥ 95% em cnpj_lib/.
05

Artefato A · API HTTP

A
app/ · FastAPI sobre SQLite read-only
deps: fastapi · pydantic · uvicorn esforço: M porta: 127.0.0.1:8000 auth: nenhuma (local)

Conexão por request via URI file:dados_cnpj.db?mode=ro&immutable=0. WAL permite N readers concorrentes mesmo durante import do pipeline. Lookups (6 tabelas, ~200k linhas, ~30 MB) carregam em memória no startup via @lru_cache.

Modelagem JSON de GET /cnpj/{cnpj}

Decisões transversais:

  • CNPJ fragmentado — expõe completo, formatado e objeto partes. Cliente escolhe.
  • Códigos — estrutura uniforme {codigo, descricao}, mesmo para enums sem lookup no BD (descrição vem de cnpj_lib/dominio.py).
  • Datas — converte YYYYMMDD para ISO 8601; "00000000" vira null.
  • capital_social — parseia string para Decimal; agrega _formatado opcional.
  • CPF mascarado — expõe valor como vem (***349729**) + flag mascarado: true.
  • Resposta principal — somente counts de filiais e sócios; listagem via subrotas /cnpj/{basico}/estabelecimentos e /cnpj/{basico}/socios.

Artefato I · MCP Server

I
mcp_server/ · FastMCP via stdio
deps: mcp[cli] ≥ 1.27 esforço: P importa: app.servicos · não usa HTTP

Tools importam serviços do FastAPI diretamente — sem rede, sem subprocess, mesma transação. Cada tool de listagem aceita limit (default 50, máx 500) e offset; retorna {total, retornados, tem_mais, lista}. buscar_empresa nunca retorna listas brutas — só counts.

09

O que não fazer agora

  • Não exportar para a internet pública. O briefing Nº 001 já registrou: busca reversa por nome de sócio PF é zona LGPD vermelha. Se um dia abrir, será com auth e sem indexar pessoa física.
  • Não construir snapshot histórico para delta_mensal no MVP. Diff real exige preservar safras anteriores — fora do escopo das três semanas e meia. MVP retorna apenas metadados de controle_importacao.
  • Não migrar database.py para SQLAlchemy. Schema é fixo, queries são poucas; ORM adiciona dependência e import time sem benefício. Repositórios finos em sqlite3 puro bastam.
  • Não criar UI web no Caminho 01. O briefing é "caixa-preta pra mim" — API e MCP. Dashboard visual é Caminho 02 (Observatório de deltas).
  • Não tentar implementar tools MCP que requerem mais de 500 linhas em uma única resposta. Paginação manual é parte do contrato; tools que não couberem nesse padrão são resources, não tools.
