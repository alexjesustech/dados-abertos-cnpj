06

Decisões registradas que continuam válidas

Pontos definidos no Briefing Nº 001 e confirmados na execução do Caminho 01. Se a decisão for revisitada, ela deve ser renegociada explicitamente — não simplesmente abandonada.

Decisão Por quê continua válida
Espelho local em SQLite, sem rate-limit, em 127.0.0.1 Uso pessoal. Tirar de loopback exige auth + rate-limit + observabilidade pública — Caminho 04, não 01.
SQL puro nos repositórios; sem SQLAlchemy Schema é fixo (definido pela RFB), queries são poucas e curtas. ORM adiciona indireção sem ganho.
Paginação manual com limit/offset + tem_mais MCP não tem paginação nativa para tools/call. Padrão é replicado em todas as 9 tools.
Lookups em memória, carregados na lifespan da API ~200 k linhas, ~30 MB RAM. Tradução O(1) em vez de JOIN por request — economiza muito tempo de resposta.
synchronous = OFF no pipeline Aceita crash de OS como custo conhecido em troca de velocidade de ingestão. Pipeline é re-executável; banco é descartável.
07

O que não fazer ao retomar

  • Não expor a API publicamente sem auth. Default é 127.0.0.1:8000 por uma razão — auth + rate-limit ficam no Caminho 04, fora do escopo atual.
  • Não tentar diff de safras com snapshot atual. delta_mensal está em MVP (só metadados); diff real exige preservar tabelas históricas — pertence ao Caminho 02.
  • Não migrar database.py para SQLAlchemy. Schema é fixo, queries são poucas. Repositórios em sqlite3 puro bastam.
  • Não fazer busca reversa por nome de sócio PF indexável. O Briefing Nº 001 marcou como zona LGPD vermelha — Casa dos Dados sofre pedidos recorrentes.
  • Não tentar tools que retornem mais de 500 linhas em uma única chamada. Disciplina de paginação é parte do contrato — tools que não caibam viram resources, não tools.
  • Não expor o dashboard do monitor na LAN. Os endpoints /api/run e /api/stop não têm auth — qualquer host na mesma rede consegue disparar/matar o pipeline.
